## Context

In 2026, AI coding tools have become essential in every developer's daily workflow — from small bug fixes to large features, from greenfield projects to legacy codebases.

The term "vibe coding" has entered the mainstream: type a prompt, and the AI agent handles the rest. LLMs grow more capable by the day, and can now implement most technical tasks autonomously.

This has fundamentally shifted the developer's role. We spend less time writing code and more time directing, reviewing, and validating AI-generated output. The bottleneck is no longer typing speed — it's clarity of intent. AI is remarkably good at *how* to build something, but it still depends on humans to define *what* to build and *why*.

## Problem Statement

However, faster output doesn't mean correct output. More code is produced, but quality control hasn't kept up. In practice, several recurring problems emerge when teams rely heavily on AI coding agents.

### Redundant loop

Without a clear spec, developers fall into a costly refinement cycle — prompting, reviewing, correcting, and re-prompting. Each iteration burns tokens and time because the AI lacks a precise definition of "done."

Example with a button:
- **prompt:** create a button in the center
- **ai:** creates it too big with ugly centering
- **prompt:** no, I need...

This back-and-forth can repeat dozens of times for a single feature. A clear spec upfront eliminates most of these rounds.

### Token consumption

When implementing a new feature, the AI needs to read the entire repository to understand existing behavior before it can reason about the new requirement. But most codebases are poorly structured for machine consumption — outdated patterns, implicit conventions, and scattered logic force the AI to ingest far more context than necessary. This leads to heavy token consumption with diminishing returns.

### Code is a liar

Code often does more than what it was designed for — dead code, redundant complexity, and undocumented side effects accumulate over time. When the AI reads this code as its source of truth, it can hallucinate intent, replicate bad patterns, and produce subtly wrong results. A specification, by contrast, captures *intended* behavior rather than *accidental* behavior.

## SDD Solution

SDD (spec-driven design) concept isn't new, it was first introduced by Microsoft in the 2000s. The goal is merely writing the requirement specification in detail before writing the code.

And the most common example is API first, where we write the API spec following the OpenAPI spec and the team follows it strictly to write down the implementation.

In 2026, the SDD method is emerging again. SDD has come back because in 2026 we mostly focus on writing and clarifying the requirements with the strong support of AI coding agents.

But SDD in 2026 has changed a lot because we may have greenfield and brownfield projects, and adopting SDD isn't only about specifying the requirements first but also about how to respect the specification and keep track of it to grow in the long term.

To answer this question, we will go through 3 kinds of SDD levels and let's see an example support tool to achieve this goal.

### 3 Kinds of SDD

#### Spec-First

In spec-first development, a specification is written before implementation to guide the initial development.
Once code exists, the spec may or may not be maintained.
This one is suitable for prototypes and where the code isn't maintained in the long term.

#### Spec-Anchored

The spec is maintained alongside the code throughout the system's lifecycle. Changes to behavior require updating both the spec and the code, keeping them synchronized.

The team must commit to updating the specs whenever behavior changes. This level fits almost all of the modern SDLC when we have AI agents that can work well on almost all tasks. It provides the benefit of clear documentation and verifiable requirements without demanding to read all of the code.

The heart of this method is building the tests following the spec strictly and testing against the source code to ensure the spec and code remain aligned.

#### Spec-as-Source

The spec is the only artifact humans edit directly. Code is entirely generated from the spec and should never be manually edited. Changes to the spec trigger code regeneration.

If your code generation tool is mature and well tested on the input tasks (not actually good for experimental tickets that require human in the loop, like needing to execute multiple commands to get data from where the AI agent isn't allowed to read,...), choosing this level is good.

Like spec-anchored, automated testing is required even more strictly than spec-anchored if we go with this method. Not only the tests but also unit tests and integration tests to ensure the code works, and automation tests to ensure the product of the code works as expected.

A higher level doesn't always mean better; choosing the right level and step-by-step upgrading it is a good move. You cannot jump to spec-as-source if your AI coding tool isn't mature, but moving forward to a higher level will be a good move.

## A Typical SDD Workflow

The typical SDD workflow with an AI agent starts with defining the constitution, where you work with the AI to define the mission, tech stack, and roadmap. Then we go into the development phase, step by step introducing new features. And the most important process of modern SDD is the replanning phase.

![Pipeline Flow](/images/sdd/workflow.png)

### Constitution

Constitution is the process of defining the high level for your project. For greenfield projects, we only need to determine the high-level goal (concisely for your project) and the tech stack and roadmap, but for brownfield, we may need to ask the agent to read the whole codebase or related documentation to build the first specification for your project to ensure the LLM tool can understand the purpose of your codebase in future sessions without reading all of the codebase.

The outcome of this step will be documentation (e.g., markdown) that lives together with your project (cohesive) to facilitate the LLM tool to consume it next time.

### Develop New Feature

To develop a new feature, we will go through multiple steps.

**Specification:** Define a good specification, which includes clarifying to remove ambiguous points and make the spec ready for implementation.
In this step, the LLM agent will read the old specs following the constitution to clarify the requirements.
This answers what to build. We will follow the Gherkin format (like BDD) for acceptance criteria to make the spec more deterministic on the outcome. This is crucial for the agent to implement the code to fulfill it, because talking around without clarifying the storage is a trap for the AI coding tool.

**Planning:** Write the implementation plan for the clarified requirements. The implementation plan will stop at a high level (or maybe lower at code level depending on what the team wants to follow).
This answers how to build it at a high level.

**Implementation:** Break the task into smaller verifiable tasks, and then feed them into the AI agent to write the code. Writing the code will include writing tests.

**Validation:** Run the tests to ensure the code meets the requirements. If the tests fail, there will be 2 options: updating the code or correcting the spec. It will be a human decision.

### Replanning

Replanning is an important action when the AI needs to update the spec after the implementation because there is no perfect implementation plan.
We may need to change something once hands-on with the actual code to fulfill the checklist items.
The need for this phase is painful but in actual software development we usually have it. Therefore we need to plan to adapt these changes to keep the spec synchronized with the code.

This phase can be triggered automatically by the AI and gets the approval of a human because it changes and refines the spec which was aligned with the human.

In practice, this can also happen after code review when we have something that changes in terms of product and tech (not a good principle), so that we need to update the constitution and the previous specs.

This replanning is also one of the really hot topics recently: harness engineering.

## Tools

When talking about spec kit in April 2026, we have 4 popular tools: Kiro by AWS, OpenSpec by Fission-AI, BMAD, and spec-kit by GitHub.

| Tool | Best for | Complexity | Workflow style | Key trade-off |
|------|----------|------------|----------------|---------------|
| Spec Kit | Greenfield (new) projects | Medium | Structured phases (specify → plan → tasks → implement) | Thorough checkpoints, but adds overhead for small changes |
| BMAD | Large-scale enterprise / complex systems | High | Multi-agent (PM, Architect, QA, and other specialized roles) | Comprehensive but resource-heavy and steeper learning curve |
| OpenSpec | Existing / brownfield codebases | Low | Lightweight iterative (propose → apply → archive) | Fast and minimal, but less formal structure |
| Kiro | AWS / Claude-centric teams | Medium | Integrated IDE + built-in spec pipeline | Excellent structure and cloud integration, but vendor (AWS/Bedrock) lock-in |

Most of tools here support different AI coding tool (claude, codex,...). It's eliminated the problem of vendor lock.

## Conclusion

- Spec-first for AI-assisted initial development; spec-anchored for long-lived production systems; spec-as-source only when generation tooling is mature and trusted.
- Traditional design documents are advisory—developers read them, then write code that hopefully matches. SDD specs are enforced—tests fail if code diverges, and in spec-as-source approaches, code is regenerated rather than manually edited.
- We will go sustainable with using the SDD tool but adopting a new tool require step by step and deeply understand the workflow and the mindset to respect and gain benefit from it.
- SDD is like TDD and BDD, when encourage the developer to focus about the end user behavior and initial goal for the coding instead of working around.
