# Problem Statement

* Redundant loop
With using vibe coding, we cannot finish the task in a one single prompt. Human will fall into ta loop of refining the requirement to refine the feature. 

Example with the button:
- prompt: create a button in the center
- ai: create too big with urgly center
- prompt: no, I need...

* Token consumption
- When implementing a new feature the AI need to read the whole code repository to just understand the existing features, but the actual code is usually outdated and not structure well to consume quickly. 
So AI will read all of the code to know the existing features to clarify new feature.  
It needs to token consuming.

* Code is a lier
- Most of the code do more than what it was designed for (deadcode, redundant complex, unknown behavior,...), so reading the code can makes the LLM halucinate and lead to wrong result

# SDD solution
SDD (spec driven design) concept isn't new, it has been first introduced by Microsoft in 2000s, it's not a new one. The goal is merely writing the requirement specication detailedly before writing the code.

And the most common example is API first, when we write the API spec follow the Open API spec and the team follow it strictly to write down the implementation.

In 2026, SDD method is emergedly again. SDD come back because 2026 we mostly focus on writing and clarifying the requirement with the strong support of the AI coding agent.

But the SDD is 2026 is changed a lot because we may have the greenfield and brownfield projects, the adopting SDD isn't only sepcify the requirement first but how to respect the specification and keep track it to grow in long term. 

To answer this question, we will go through 3 kinds of SDD level and the let's an example support tool to achieve this goal.

## 3 kinds of SDD
### Spec first
IN spec first development, a specifiction is written before to guide the initial implementation.
Once code exist, the spec may or may not be maintained.
This one is suitable for prototye and where the code isn't maintained in long term.

### Spec-Anchored
the spec is maintained along side the code throughout the system's lifecycle. Changes to behavior require updating both the spec and the code, keeping them synchronized. 
Team must commit to updating the specs whenever behavior changes. This level fit almost all of the modern SDLC when we have ai agent can work well on almost of the task. It provide the benefit of the clear documentation and verifiable requirements with demanding to read all of code.
The heart of this method is buidling the test follow strictly the spec and test again the source code to ensure the spec and code remain aligned.

### Spec-as-source
The spec is the only artifact humans edit directly. Code is entirely generated from the spec and should never be manaually edited. Change to the spec trigger code regenerating
If your code generation tool is mature and well tested on the input tasks (not actually good in experiment tickets that require human in the loop like need execute multiple commands to get data from where the AI agent isn't allowed to read,...), choosing this level is good.
Like the spec-anchored, automation test is required more strictly than spec-anchored if we go with this method. Not only the tests but also the unittest and integration test to ensure the code work. and automation test to ensure the product of the code work as expected.

Higher level doesn't mean always good, choosing the right level and step by step ugrading it is a good move. You cannot come to spec as source if your AI coding tool isn't mature but moving forwarding to a higher level will be a good move.

# A typical SDD workflow
The typical SDD workflow with AI agent will be started with defining the constitution when you work with the AI to define the mission, tech stack and roadmap. And then we will go into the development phases with step by step introducing a new feature. And the the most importent process of modern SDD is the replanning phase

![Pipeline Flow](/images/sdd/workflow.png)
## Constitution
Constitution is process of defining the high level for your project. For greenfield project, we only need to determine the high level goal (concisely for your project) and the tech stack and roadmap but for brownfield, we may need to ask the agent to read the whole code base or related documentation to build the first specification for your project to ensure the LLM tool can understand the purpose of your code base in the next working times without reading all of the codebase.

The outcome of this step will be the (i.e markdown) that live together with your project and live with your project (cohesive) to facilitate the LLM tool to consume it next time.

## Develop new feature
To develop a new feature, we will go through multiple steps. Step one is defining a good specification which include clarify to remove ambigious points and make the spec is read for implementation.
This step the LLM agent will read the old specs follow the constitution to clarify the requirements. 
Answer what to build. We will follow Gherkin format (Like BDD) to acceptance criteria to make the spec more deternimistic on the outcome of the spec. Which is crucial for the agent to implement the code to fulfill it. because talking around without clarifying the storage is a trick for the AI coding tool.

Next it will be writing the implementation plan for the clarified requirements. the implementation plan will stop at high code level (or maybe lower level at code level depend on what the team want to follow)
This one answer how to build it at high level. 

Implement phase: We will break out the task into smaller tasks that verifible. and then feed it into the AI agent to writing the code for it. The writing code will include writing tests.

Validation: This phase include running the test to ensure the code meet the requirement. If the test fail, there will be 2 options updating the code or correcting the spec, it will be human decision.

## Replanning
Replanning is an important action when AI need to update the spec after the implementtion because there is not perfect implementation plan.
We may need to change something once hands on with the actual code to fullfile the check list item. 
The need for this phase is pain but in actual software development we usually have it. Therefore we need to plan to adapt this changes to keep the spec synchronized with the code. 

This phase can be triggered automatically by the AI and get the approval of human because it changes and refined the spec which was aligned with the human. 

In practical, This one can happen after the code reviewing also when we have something changes in term of product and tech (not good principle) so that we need to update the constitution and the spec of the previous spec.

This replanning is also one of the really hot topic recently: harness engineering.

# Conclusion
- Specfirst for AI-assisted initial development; spec-anchored for long-lived production systems; spec-as-source only when generation tooling is mature and trusted.
- Traditional design documents are advisory—developers read them, then write code that hopefully matches. SDD specs are enforced—tests fail if code diverges, and in spec-as-source approaches, code is regenerated rather than manually edited.
