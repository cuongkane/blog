# The Problem

You have a Kafka topic receiving millions of user events per second. You need to:

1. **Deduplicate** events (same event ID must not be processed twice)
2. **Enrich** each event with business rules that can change at runtime (via a config stream)
3. **Write results** to an output Kafka topic with **exactly-once** guarantees

The pipeline looks like this:

```
Kafka "user-events" (8 partitions)
    → KafkaSource
    → Dedup operator (keyed state: has this event been seen before?)
    → Enrich operator (broadcast state: live config rules)
    → KafkaSink
Kafka "processed-events"
```

This post follows a single message through this pipeline — from the moment it leaves Kafka to the moment its result is committed — and explains what happens at every physical layer along the way.

# Flink Components in 60 Seconds

Before we trace our message, let's name the moving parts. In Flink, there is a clear hierarchy from the physical machines all the way down to the threads that process your data:

**JobManager** — The brain. A single process (one K8s Pod) that coordinates the entire job. It decides where tasks run, triggers checkpoints, and handles failures. It contains three internal components: the **Dispatcher** (receives job submissions), the **ResourceManager** (negotiates slots from TaskManagers), and the **JobMaster** (manages the execution graph and checkpoint coordination for a specific job).

**TaskManager** — The workers. Each TaskManager is a separate JVM process (one K8s Pod) that registers with the JobManager and offers its computational capacity. A typical production cluster has many TaskManager Pods spread across Kubernetes nodes. Each TaskManager has a fixed amount of memory and CPU, configured at Pod creation time.

**Slot** — A logical resource container *inside* a TaskManager. Not a process, not a thread — just a budgeted slice of the TaskManager's resources. If a TaskManager has 10 GB of memory and 2 slots, each slot gets roughly half the managed memory. Slots are what the JobManager "books" when it needs to run tasks.

**Operator** — A processing step in your pipeline (i.e `KafkaSource`, `Dedup`, `Enrich`, `KafkaSink`). An operator defines *what* to do with data (map, filter, aggregate, etc.), but by itself it doesn't run — it needs to be instantiated as subtasks.

**Subtask** — A parallel instance of an operator. If `Dedup` has parallelism=8, there are 8 Dedup subtasks: `Dedup[0]` through `Dedup[7]`. Each subtask runs in a slot and processes a portion of the data. A subtask is the actual unit of execution — it's a Java thread that reads records, calls your function, and writes output.

# The Cluster on Kubernetes

Now let's see how this maps to a real K8s deployment with **parallelism=8** and **2 slots per TaskManager**.

We need 8 slots total. With 2 slots per TaskManager, that's **4 TaskManager Pods**. The Flink Kubernetes Operator (or native K8s integration) creates these as a Deployment or StatefulSet:


![Flink Architecture on K8S](/images/flink_operation/flink_architect.png)

A few things to notice:

- **Each TaskManager Pod is one JVM process.** Kubernetes manages its lifecycle — if a Pod dies, K8s (or the Flink Kubernetes Operator) can restart it. The 2 slots inside it are purely logical: they share the JVM heap and network buffers, but each slot gets its own **isolated managed memory** budget for RocksDB. Slot 0 cannot borrow memory from Slot 1, even though they're in the same Pod.

- **8 Kafka partitions, parallelism 8, 8 slots** — each slot reads exactly one Kafka partition. This is the ideal 1:1 mapping. (If you had 16 partitions with parallelism 8, each slot would read 2 partitions. If you had 4 partitions with parallelism 8, 4 slots would sit idle — Flink never splits one Kafka partition across multiple readers.)

- **Slot sharing** means all four operators run inside the **same slot** for the same subtask index. Slot 0 runs Source[0], Dedup[0], Enrich[0], and Sink[0] — all chained into a single Java thread. Data flows between them via direct method calls, not the network.

- **RocksDB state lives on ephemeral disk** (the Pod's local storage). It disappears when the Pod is terminated. But that's fine — the checkpoint data is on S3 (durable). On recovery, a new Pod downloads state from S3 and rebuilds its RocksDB.

- **Scaling up** means increasing parallelism and adding more TaskManager Pods. Going from parallelism=8 to parallelism=16 means adding 4 more TM Pods (8 more slots), doubling Kafka partitions to 16, and taking a savepoint to redistribute state. Kubernetes handles the Pod scheduling; Flink handles the state redistribution across the new key group ranges.

# Chapter 1: A Message Is Born

A user event arrives in Kafka partition 3 of the `user-events` topic:

```json
{
  "event_id": "evt-88821",
  "user_id": "user-42",
  "action": "purchase",
  "amount": 59.99,
  "timestamp": 1712100000000
}
```

Kafka stores it at **partition 3, offset 50701**.

# Chapter 2: Flink Picks It Up

Source[3] is a **subtask** — the 4th parallel instance of the KafkaSource operator. It's running inside **Slot 3 in TaskManager Pod 2** on Node A. Its background fetcher thread continuously calls Kafka's `poll()`. Our message arrives in a batch of records. The fetcher places it into a hand-over queue, and the slot's task thread picks it up.

The source deserializes the raw Kafka bytes into a Flink `StreamRecord` — now it's a Java object living in the JVM heap of TaskManager Pod 2.

**Important:** Flink does NOT use Kafka's consumer group offsets for recovery. It stores offsets in its own state. Kafka's consumer offsets may be committed optionally for monitoring, but Flink ignores it during failover.

# What Our Message Sees Around It in the Stream

Our message is not alone. As it travels through the pipeline, it flows alongside two special markers. Your business logic never sees them — Flink's runtime handles them behind the scenes — but they're critical to understanding how Flink keeps time and makes snapshots.

```
... [record] [record] [WATERMARK] [record] [OUR MESSAGE] [BARRIER cp-43] [record] ...
```

### Watermarks — "When Can We Close a Time Window?"

**The problem watermarks solve:** Imagine you have a 5-minute window aggregation — "sum all purchases between 12:00 and 12:05." Events arrive out of order (network delays, retries). How does Flink know when 12:00–12:05 is "complete" and it's safe to emit the result?

That's what watermarks do. A watermark is a timestamp that flows through the stream and means: **"all events with timestamps before this value have already arrived."** When a watermark of 12:05 passes through the window operator, the operator knows the 12:00–12:05 window is complete and can emit its result.

**Who generates them?** Each **source subtask** generates watermarks based on the event timestamps it sees in its Kafka partition. Source[3] looks at the timestamps of the records it reads, and periodically emits a watermark saying "I've seen everything up to time T."

**Who tracks them?** Each **subtask** (not the JobManager, not the TaskManager — the subtask itself) tracks watermarks from all its input channels. When a subtask has multiple inputs (like Dedup[4] receiving from 8 sources after keyBy), it takes the **minimum** watermark across all inputs. The subtask's watermark only advances when the slowest input advances. One straggling Kafka partition holds back all downstream time-based operations.

In our pipeline, the Dedup operator doesn't use time windows, so watermarks flow through it without triggering anything. But if we added a windowed aggregation, watermarks would be essential.

### Checkpoint Barriers — "Take a Snapshot at This Exact Point"

**The problem barriers solve:** Flink needs to take a consistent snapshot of the entire distributed pipeline — all 8 subtasks, across 4 Pods — without stopping processing. Barriers are how it coordinates this.

**Who triggers them?** The **JobManager's CheckpointCoordinator** sends a "start checkpoint 44" signal to all source subtasks.

**Who injects them?** Each **source subtask** injects a barrier into its output stream at the current position. The barrier divides the stream: everything before it belongs to checkpoint 44, everything after belongs to the next checkpoint.

**Who handles alignment?** Each **subtask** handles its own barrier alignment. When Dedup[4] receives barrier 44 from Source[0] but not yet from Source[5], the subtask itself blocks the fast channel and keeps processing the slow one, until all 8 barriers arrive. Then it snapshots its state and forwards the barrier downstream. No central coordinator is involved in this per-subtask alignment — each subtask does it independently.

Our message is at offset 50701. Checkpoint 43's barrier was injected earlier (at offset 50600). Our message arrived after that barrier, so it belongs to checkpoint 44's epoch. Its effects will be captured in checkpoint 44's snapshot.

# Chapter 3: The Dedup — Does This Message Need a Network Hop?

This is where the pipeline design matters. The Dedup operator is keyed by `event_id` — it needs to check "have I seen `evt-88821` before?" in its keyed state.

Since the pipeline uses `keyBy(event -> event.event_id)` before the Dedup operator, Flink must route each record to the subtask that **owns** that key's state. The routing works like this:

```
"evt-88821"
  → hashCode() → MurmurHash → modulo 128 → key group 71
  → key group 71 × 8 / 128 → subtask 4
```

Our message was read by Source[3] in Pod 2 (Node A), but it needs to go to **Dedup[4] in Pod 3 (Node B)**. This means a **network shuffle** — the message must physically cross Kubernetes nodes.

### What Happens on the Wire

1. **Serialization:** Source[3]'s thread serializes our message into bytes and writes them into a 32 KB network buffer (a `MemorySegment` from the pre-allocated network pool).

2. **Partitioning:** The `ResultPartition` for Source[3] has 8 sub-partitions (one per downstream Dedup subtask). Our message goes into sub-partition 4.

3. **TCP transport:** Since Dedup[4] lives in a different Pod on a different K8s node, the buffer is sent via **Netty over TCP** from Pod 2 (Node A) to Pod 3 (Node B). The bytes travel through the OS TCP stack, across the Kubernetes network (CNI plugin, potentially across physical switches), and arrive at Pod 3's Netty server.

4. **Deserialization:** Dedup[4]'s input gate receives the buffer, deserializes it back into a `StreamRecord`, and hands it to the Dedup operator.

If the key had hashed to Dedup[3] instead (same Pod as Source[3]), the transfer would be a memory-to-memory handoff within the same JVM — no TCP, no K8s network. But the record would still be serialized/deserialized because the `keyBy` breaks operator chaining (source and dedup run in different threads).

### Backpressure

What if Dedup[4] is slow? The receiving slot's input buffers fill up. It stops granting "credits" to the sender. The sender blocks because it has no credits to send. This blocking propagates backward: Source[3] eventually cannot emit records, which slows down its Kafka `poll()` rate. No data is ever dropped — the entire pipeline slows down together.

# Chapter 4: The Dedup Checks State — RAM or Disk?

Our message has arrived at **Dedup[4]** — the 5th parallel instance of the Dedup operator, running in **Slot 4 inside TaskManager Pod 3** on Node B. The operator needs to check: has `evt-88821` been seen before?

To understand where that answer comes from, we need to look at how the Pod's memory is carved up and how RocksDB physically stores state.

## How the TaskManager Pod's Memory Is Organized

When TaskManager Pod 3 starts, its total memory (10 GB in this example — you configure this via the Pod's resource request; it could be 4 GB for a light job or 32 GB for a heavy one) is divided into strictly separated regions before any task runs:

```
TaskManager Pod 3 (10 GB total in this example)
│
├── JVM Heap
│   ├── Framework Heap (~400 MB)
│   │   Flink's internal machinery: RPC, classloaders, logging.
│   │
│   └── Task Heap (~2 GB)
│       Your user-defined functions' Java objects and broadcast state.
│       Garbage-collected normally by the JVM.
│
├── Network Memory (~1 GB) — off-heap, pre-allocated at startup
│   Fixed-size 32 KB buffer segments for records traveling between
│   operators over the network. SHARED across both slots in this Pod.
│
└── Managed Memory (~5 GB) — off-heap, outside the JVM's control
    Raw OS memory that the JVM doesn't know about. Flink tracks it
    manually with atomic counters. This is where RocksDB lives.
    HARD-PARTITIONED: Slot 4 gets ~2.5 GB, Slot 5 gets ~2.5 GB.
    They cannot borrow from each other.
```

The proportions are configurable. Flink's default splits roughly 40% for managed memory, 10% for network, and the rest for JVM heap. The bare minimum for a TaskManager Pod is around 1.5–2 GB (enough for framework overhead plus a small heap and minimal managed memory), but production workloads with RocksDB typically need 4 GB or more per Pod to give the block cache enough room.

The critical point: **managed memory is hard-partitioned per slot**. Slot 4 and Slot 5 each get their own independent memory manager with their own budget. If Slot 4's RocksDB needs more cache space, it cannot steal from Slot 5. This prevents one hot subtask from degrading its neighbor.

Network memory, by contrast, is soft-shared — both slots draw from the same global pool, and buffers flow dynamically to wherever demand is highest.

## How RocksDB Uses the Slot's Managed Memory

Slot 4's 2.5 GB of managed memory is further divided by consumer weights. In a typical streaming job with no batch operators or Python UDFs, the state backend (RocksDB) gets about half:

```
Slot 4's Managed Memory: 2.5 GB
│
└── RocksDB budget: ~1.0 GB (allocated to all RocksDB instances in this slot)
    │
    ├── Write Buffer Manager: ~100 MB (10%)
    │   The in-RAM memtable where all new writes land.
    │   Shared across Dedup[4]'s RocksDB AND Enrich[4]'s RocksDB.
    │
    └── Block Cache: ~900 MB (90%)
        LRU cache for SST data blocks read from disk.
        Also shared across both RocksDB instances in this slot.
        │
        ├── High Priority Pool: ~90 MB (10% of cache)
        │   Pinned index blocks and bloom filter blocks.
        │   Never evicted. This is what makes key lookups fast
        │   without touching disk.
        │
        └── Default Pool: ~810 MB (90% of cache)
            Data blocks containing your actual key-value pairs.
            Evicted least-recently-used when full.
```

**This is the sharing boundary that matters:** Dedup[4] and Enrich[4] are two separate RocksDB instances (separate databases, separate files on disk), but they **share** the block cache and write buffer within Slot 4. They compete for the same 900 MB of cache. But they never compete with Slot 5's RocksDB — that's a completely separate memory budget.

## RocksDB's LSM-Tree: How State Lives on RAM and Disk

RocksDB uses a Log-Structured Merge-Tree (LSM-tree). This data structure is why writes are fast, reads are usually fast, and state can grow far beyond available RAM. Let's trace our message through both paths — first the read (the dedup check), then the write (marking it as seen).

### The Read Path — Checking "Have I Seen This Before?"

Dedup[4] needs to look up "evt-88821" in its state. RocksDB searches through several layers, from fastest to slowest:

**Step 1 — Memtable (RAM).** RocksDB first checks the active memtable — a sorted in-memory structure holding the most recent writes. If "evt-88821" was written in the last few seconds, it's here. Cost: nanoseconds.

**Step 2 — Block cache (RAM).** If not in the memtable, RocksDB checks the block cache. SST files on disk are divided into small blocks (4 KB each). When a block is read from disk, it's cached in the block cache. If the block containing our key was read recently (perhaps when checking a nearby key), it's still in cache. Cost: microseconds.

**Step 3 — Bloom filters (RAM).** If not in the cache, RocksDB needs to search SST files on disk. But it doesn't read every file blindly. Each SST file has a **bloom filter** — a compact probabilistic data structure that can definitively say "this key is NOT in this file." Bloom filters are loaded into the high-priority pool of the block cache when the SST file is opened, and they stay pinned there permanently (never evicted). RocksDB checks the bloom filter for each SST file, and typically eliminates 99%+ of files without touching disk at all.

**Step 4 — Disk read (SSD).** For the one or two SST files whose bloom filters say "maybe" (bloom filters have no false negatives but can have false positives), RocksDB reads the specific 4 KB data block from disk. The block is loaded into the block cache (so future reads of nearby keys are fast), then searched for our key. Cost: a few milliseconds for one SSD read.

```
Read path for "evt-88821":

1. Memtable (RAM)     → miss
2. Block cache (RAM)  → miss (first time seeing this key)
3. Bloom filters (RAM, pinned permanently)
   → 000001.sst: "definitely NOT here" (skip)
   → 000002.sst: "definitely NOT here" (skip)
   → 000003.sst: "maybe here" (must check disk)
4. Read one 4 KB block from 000003.sst into block cache
   → key not found → first occurrence!
```

For dedup with a short time window (most duplicates arrive within seconds), the block cache hit rate is 95%+. The very first lookup for a new key is the slowest (may require a disk read), but once the data block is loaded into the cache, all subsequent lookups for keys in that block are RAM-speed.

### The Write Path — Marking "evt-88821" as Seen

Now that we know the key is new, Dedup[4] writes "evt-88821 → true" to state. The write path is designed so your processing thread never waits on disk:

1. Flink serializes the key (with a key-group prefix) and the value into bytes, and adds them to a **write batch** — a buffer that accumulates multiple writes
2. When the batch is flushed, all entries go into the **memtable** — a sorted in-memory structure (a skip list). This is pure RAM. Cost: microseconds. Your processing thread returns immediately.
3. That's it for your thread. Everything below happens in the background.

### How State Moves from RAM to Disk (Background)

The memtable doesn't grow forever. Here's the background flow that moves data to disk without ever blocking your processing:

1. **Memtable fills up** (reaches its size limit, e.g., 50 MB). RocksDB marks it as **immutable** — no more writes go here — and creates a fresh empty memtable for new writes.
2. A background **flush thread** writes the immutable memtable to disk as a new **SST file** (Sorted String Table) at Level 0. The SST file is sorted by key and immutable — once written, it never changes. This flush happens asynchronously; your processing thread is not blocked.
3. Over time, Level 0 accumulates multiple SST files. Background **compaction threads** merge them: read several L0 files, sort-merge their entries, remove deleted or overwritten keys, and write the result as new SST files at Level 1. Then L1 files compact into L2, and so on.
4. Each level is roughly 10x larger than the previous, creating a hierarchy that efficiently organizes terabytes of state:

```
The data flows downward over time:

  Memtable (RAM, ~50 MB)
      ↓ flush (background)
  Level 0 (disk, up to ~256 MB, recently flushed SSTs)
      ↓ compaction (background)
  Level 1 (disk, ~256 MB, sorted, non-overlapping SSTs)
      ↓ compaction (background)
  Level 2 (disk, ~2.5 GB)
      ↓
  Level 3 (disk, ~25 GB)
      ↓
  ...can grow to hundreds of GB per subtask
```

The key insight: **your processing thread only touches the memtable (RAM)**. Flushing and compaction happen in dedicated background threads managed by RocksDB (typically 2 per operator). This is why Flink can sustain millions of state writes per second even with terabytes of total state on disk.

### SST Files on Disk

Each operator subtask's RocksDB lives in its own directory on the Pod's ephemeral storage:

```
/tmp/flink/job_{jobId}_op_dedup_uuid_{random}/db/
├── 000001.sst       ← Level 1 SST, ~64 MB, sorted, immutable
├── 000002.sst       ← Level 1 SST
├── 000003.sst       ← Level 2 SST (produced by compaction)
├── 000004.sst       ← Level 0 SST (recently flushed from memtable)
├── MANIFEST-000005  ← Metadata tracking which SSTs exist at which level
├── CURRENT          ← Points to the current MANIFEST
└── OPTIONS-000006   ← RocksDB configuration snapshot
```

Each Flink state descriptor (like the ValueState "seen" in Dedup) becomes a separate **column family** in the RocksDB instance. Column families have their own SST files and compaction schedules, but share the block cache and write buffer within the slot.

**SST files are immutable.** Once written, never modified. New data creates new SSTs. Compaction reads old SSTs, merges them, writes new ones, deletes the old ones. This append-only design is what makes checkpointing fast: to create a point-in-time snapshot, Flink just creates **hard links** to the current SST files. Even if compaction deletes the originals afterward, the hard links keep the data alive until the snapshot is uploaded to S3.

## The Dedup Verdict

Back to our message. The read path found no match — "evt-88821" has never been seen. The write path records "evt-88821 → true" in the memtable in microseconds. The message passes downstream. If the same event ID arrives again later, the read path will find it (in the memtable if recent, or in the block cache after the data is flushed to an SST), and the duplicate will be silently dropped.

# Chapter 5: The Enrich — Broadcast State

The message moves from Dedup[4] to Enrich[4]. Since they're in the **same slot** and there's no keyBy between them, they are **chained** — they run in the same thread. When Dedup[4] emits a record, it directly calls Enrich[4]'s processing function as a plain Java method call within the same thread. The record is not serialized into bytes, not written to a network buffer, not copied — the same Java object in memory is simply passed from one function to the next. This is as fast as any normal function call in your code: nanoseconds.

The Enrich operator needs to look up the current business rules. These rules come from a separate Kafka topic (`config-updates`) and are delivered to **all** parallel instances via **broadcast state**.

### How Broadcast State Works

Unlike keyed state (which is partitioned — each subtask holds different keys), broadcast state is **replicated** — every subtask holds the same complete copy. When a config update arrives on the config stream, Flink's `BroadcastPartitioner` sends it to all 8 subtasks. Each one updates its local copy independently.

Broadcast state lives as a plain **Java HashMap in JVM heap** — not in RocksDB. It's simple: Enrich[4] calls `broadcastState.get("pricing-rules")` which reads from an in-memory HashMap. No disk, no serialization.

Because every subtask holds an identical copy of the rules, all 8 Enrich instances apply the same logic — even though they process different keys. This is how you push live config changes to a running pipeline without restarting it.

### Why Only Config Updates Can Modify the State

When processing our user event, the operator can only **read** the broadcast state — it cannot change the config. When processing a config update from the broadcast stream, the operator can **write** to it — that's how config gets updated.

Why this restriction? User events are partitioned — Enrich[0] and Enrich[4] see different events. If both could modify the broadcast state based on different events, the 8 copies would diverge and become inconsistent. Config updates from the broadcast stream are sent to **all** subtasks, so every subtask applies the same update and stays in sync.

Our message is enriched with the current pricing rules and passed to the Sink.

# Chapter 6: The Sink — Writing to Kafka (But Not Committing Yet)

Enrich[4] and Sink[4] are also chained — same slot, same thread, direct method call. The enriched message arrives at Sink[4] with no serialization, no network hop, no buffer copy — just one Java object passed within the same thread.

### How the Subtask Writes to Kafka Directly

Each Sink subtask has its own **Kafka producer** running inside the subtask thread. Sink[4] writes directly to the Kafka broker — the JobManager is not involved in data delivery at all. The JobManager only coordinates checkpoints and failure recovery; it never touches the actual records.

All 8 Sink subtasks write to Kafka in parallel, each with its own producer. This doesn't cause conflicts because Kafka handles concurrent producers natively: each producer appends to the target partition independently, and Kafka brokers serialize the appends per-partition. If multiple Sink subtasks write to the same output partition, Kafka simply appends their records in arrival order. If they write to different partitions, they don't interact at all. There's no merge conflict — Kafka's append-only log design means producers never overwrite each other's data.

### The Transaction Keeps It Invisible

Sink[4] has an open Kafka **transaction**. When it calls the Kafka producer's send, the record is physically stored in the Kafka broker, but it is **invisible** to downstream consumers that use read-committed isolation. The Kafka transaction is still open. Our message is in limbo — written but not committed.

It will stay this way until the next checkpoint completes.

# Chapter 7: The Checkpoint — Making It All Durable

Every 30 seconds (configurable), the CheckpointCoordinator on the JobManager triggers a distributed snapshot. This is how Flink makes the pipeline's progress durable and enables exactly-once guarantees.

Here's what happens, and what our message's state looks like at each phase:

```mermaid
sequenceDiagram
    autonumber
    participant JM as JobManager
    participant Src as Sources [0-7]
    participant Op as Dedup + Enrich [0-7]
    participant Sink as Sinks [0-7]
    participant S3 as S3

    rect rgb(230, 240, 255)
        Note over JM,S3: 1. Trigger + Barrier Injection
        JM->>Src: triggerCheckpoint(44)
        Src->>Src: snapshot Kafka offsets
        Src->>Op: emit CheckpointBarrier(44) into data stream
    end

    rect rgb(240, 235, 250)
        Note over JM,S3: 2. State Snapshot (async)
        Op->>Op: barrier alignment across inputs
        Op->>Op: RocksDB hard-link snapshot (microseconds)
        Op->>Sink: forward barrier
        Op-->>S3: upload new SST files (background thread)
    end

    rect rgb(255, 235, 235)
        Note over JM,S3: 3. Sink Pre-Commit
        Sink->>Sink: flush() pending records to Kafka
        Sink->>Sink: commitTransaction()
        Sink-->>S3: upload transaction ID
    end

    rect rgb(235, 250, 235)
        Note over JM,S3: 4. Finalize
        Src-->>JM: ack(44)
        Op-->>JM: ack(44)
        Sink-->>JM: ack(44)
        JM->>S3: write _metadata (atomic commit point)
        JM->>Sink: notifyCheckpointComplete(44)
        Sink->>Sink: beginTransaction() for next cycle
    end
```

### Phase 1: Barrier Injection

The JobManager tells all 8 source subtasks: "start checkpoint 44." Each source snapshots its current Kafka offsets (Source[3] records partition-3: offset 50702 — just past our message), then injects a CheckpointBarrier into the data stream.

Remember from earlier: our message was at offset 50701, and it arrived *before* this barrier. So our message belongs to checkpoint 44's epoch. The barrier flows downstream like a regular record, but it's not delivered to your function — it's consumed by Flink's runtime to trigger snapshots.

**Processing does not stop.** Sources keep emitting records after the barrier. The barrier is just a marker — "everything before me belongs to checkpoint 44." Records after the barrier are already being processed for the *next* checkpoint's epoch.

### Phase 2: State Snapshot

When Dedup[4] receives barrier 44 from all its input channels (barrier alignment, as described earlier), it snapshots its RocksDB. Our message's state write — "evt-88821 → true" in the memtable — is part of this snapshot. The snapshot happens in two phases:

- **Synchronous (microseconds):** Flush the write batch to the memtable, then call RocksDB's native checkpoint API. This creates **hard links** to every current SST file in a snapshot directory. A hard link is just filesystem metadata pointing to the same disk blocks — no data is copied. Even if compaction deletes the original SST file later, the hard link keeps the data alive. A 500 GB state is "frozen" in milliseconds. The memtable is also flushed to a new SST, ensuring writes like our "evt-88821 → true" are captured on disk.

- **Asynchronous (background thread):** A separate thread compares the hard-linked SST files against the previous checkpoint. Only **new** SST files (those produced by memtable flushes and compaction since checkpoint 43) are uploaded to S3. Previously uploaded SSTs are referenced by metadata only. A 500 GB state that had 200 MB of new writes only uploads those 200 MB. This is incremental checkpointing — cost scales with the write rate, not the total state size.

The barrier is forwarded to the Sink immediately after the sync phase. Processing of new records (belonging to the next checkpoint epoch) resumes in parallel with the background S3 upload.

### Phase 3: Sink Pre-Commit

When the barrier reaches Sink[4]:
1. **Flush** all pending records to Kafka (our message was already sent, but any buffered-but-not-yet-sent records get flushed now)
2. **Commit** the Kafka transaction — our message and all other records in this transaction become visible to `read_committed` consumers
3. **Snapshot** the transaction ID to S3 for recovery purposes
4. **Begin** a new transaction for the next cycle

### Phase 4: Finalization

All 8 subtasks of all operators send `acknowledgeCheckpoint(44)` to the JobManager with their state handles (S3 paths). Once all 32 acknowledgments arrive (8 subtasks × 4 operators), the JobManager writes a single `_metadata` file to S3. This file is the **atomic commit point** — a checkpoint is valid if and only if its `_metadata` exists.

**Our message is now fully committed:** its Kafka offset is part of the checkpoint state, the Dedup "seen" flag is in the RocksDB snapshot on S3, and its output record is visible in the output Kafka topic.

If the entire cluster crashes after this checkpoint, Flink restores from S3 and resumes from exactly this point. Our message will not be reprocessed. No duplicates.

# What Makes This Scale?

Now that we've followed a single message through the pipeline, let's zoom out and see why this architecture handles millions of messages per second across terabytes of state.

## 1. Parallelism = Independent Lanes, Scaled by K8s Pods

Our 8 subtasks are 8 completely independent processing lanes, spread across 4 Kubernetes Pods. Dedup[0] in Pod 1 and Dedup[4] in Pod 3 never communicate with each other. They have separate RocksDB instances, separate managed memory, on separate K8s nodes.

Doubling throughput means: increase parallelism to 16, add Kafka partitions to 16, and let the Flink K8s Operator spin up 4 more TaskManager Pods. Take a savepoint, redeploy, and Flink redistributes state across the new key group ranges automatically.

The key group system makes this possible. Each key deterministically maps to exactly one subtask. No distributed locks, no cross-Pod coordination, no shared state. Each lane is self-contained.

## 2. Slot Sharing = Resource Efficiency

With 4 operators and parallelism 8, a naive approach would need 32 slots. Slot sharing puts all operators for the same subtask index into one slot — only 8 slots needed. This also means chained operators (Dedup→Enrich→Sink) pass data via direct method calls instead of network serialization, which is orders of magnitude faster.

## 3. RocksDB = State Larger Than Memory

As we saw in Chapter 4, RocksDB's LSM-tree architecture means writes always hit the in-memory memtable (microseconds), and reads are served from the block cache 95%+ of the time. State can grow to hundreds of gigabytes per subtask on the Pod's ephemeral disk, while the block cache only holds the hot working set in RAM. Bloom filters eliminate disk reads for keys that don't exist. Compaction runs in background threads, never blocking the processing thread.

Each slot's RocksDB memory is isolated — Slot 4's block cache and write buffer come from Slot 4's managed memory budget. Two operators in the same slot share this cache, but slots never share with each other. This prevents one hot subtask from degrading its neighbor.

## 4. Incremental Checkpoints = Constant-Cost Snapshots

A 500 GB state doesn't mean 500 GB uploaded every 30 seconds. The synchronous phase creates hard links (microseconds), then a background thread uploads only new SST files. With a moderate write rate, each checkpoint might upload only a few hundred MB regardless of total state size. The processing thread is paused for less than 10 milliseconds.

## 5. Exactly-Once Without Sacrificing Throughput

The combination of:
- **Source:** Flink-managed Kafka offsets (rewindable on failure)
- **State:** Checkpoint barriers create consistent distributed snapshots without stopping processing
- **Sink:** Two-phase commit with Kafka transactions (records are invisible until committed)

...means every message is processed exactly once, even across failures. And because barriers flow through the data stream rather than requiring a global pause, throughput is barely affected during checkpointing.

## 6. Broadcast State = Live Config Without Restart

The Enrich operator reads business rules from a replicated in-memory HashMap — nanosecond lookups. Config changes from the config Kafka topic are automatically pushed to all 8 parallel instances. No job restart, no redeployment, no coordination. The cost is `parallelism × config_size` of JVM heap, which is trivial for typical config data.

# Physical Boundaries Summary

Where things are shared and where they're isolated:

| What | Shared? | Where it lives |
|------|---------|---------------|
| Managed memory (RocksDB cache) between slots in same Pod | **No** — hard partition per slot | Off-heap native memory |
| RocksDB block cache between operators in same slot | **Yes** — shared within the slot | Slot's managed memory |
| RocksDB data files between operators in same slot | **No** — separate directories | Local disk |
| Keyed state between parallel subtasks | **No** — different key groups, different machines | Each subtask's own RocksDB |
| Broadcast state between subtasks | Same content, but **physically separate** copies | JVM heap per subtask |
| Network buffers between slots in same Pod | **Soft sharing** — global pool, dynamic redistribution | Off-heap pre-allocated |
| Data between chained operators | **Zero-copy** — same Java object reference | Same thread, same stack frame |
| Data between non-chained operators (same Pod) | Serialized, memory-to-memory | Network buffer pool, no TCP |
| Data between non-chained operators (different Pods) | Serialized, **Netty over TCP** | Through the K8s network |

# What Happens When Things Go Wrong

Say TaskManager Pod 3 on Node B gets OOM-killed by Kubernetes. This Pod hosted Slot 4 and Slot 5 — including our message's Dedup[4] and Enrich[4]. All state on its ephemeral disk is gone.

Here's what happens:

1. **JobManager detects the failure** via heartbeat timeout (the Pod stopped sending heartbeats)
2. Kubernetes (or the Flink K8s Operator) **schedules a replacement Pod** — possibly on a different node entirely
3. The new TaskManager Pod registers with the JobManager and offers its 2 fresh slots
4. JobManager **restores from the last completed checkpoint** (say, checkpoint 44): the new slots download RocksDB SST files from S3 and rebuild local state
5. **KafkaSources rewind** to the checkpointed offsets — they re-read messages from exactly where checkpoint 44 left off
6. **KafkaSinks abort** any uncommitted transactions (Kafka's zombie fencing kills the old producer's pending transaction)
7. Processing resumes. From the outside, it looks like a brief pause — no data lost, no duplicates

Messages that were processed between checkpoint 44 and the crash are replayed from Kafka. But because the Sink's Kafka transaction for those messages was never committed (it was part of the in-progress checkpoint 45 that didn't complete), downstream consumers with `read_committed` never saw them. When they're reprocessed, the output is identical.

This is exactly-once: Kubernetes handles Pod lifecycle (restart, reschedule). Flink handles data consistency (rewind source, restore state, abort uncommitted transactions). Together, they make the crash invisible to the rest of the system.
